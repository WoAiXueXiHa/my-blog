## 一、什么是大模型

通俗来说，大模型是：

**参数量很大、训练数据很多、能力可以泛化到很多任务的 AI 模型**

### 1. 什么是参数

可以理解成模型大脑中的**知识权重**

比如模型看到：`redis 是一种`，那么它就需要判断后面可能接`内存数据库 / 缓存 / 编程语言 / 操作系统`

模型之所以能判断，是因为它内部有大量参数，记录了训练时学到的语言规律、知识关系和上下文模式

所以：**参数不是一条条死记硬背的知识，而是模型调整出来的判断权重。**参数越多，模型通常越有能力表达复杂关系

### 2. 为什么参数多就可能更强？

小模型像一个只学过基础知识的人，他可能知道：

```text
Go 是编程语言
Redis 是缓存
MySQL 是数据库
```

但是遇到复杂问题，比如：

```text
为什么高并发系统里 Redis 和 MySQL 要配合使用？
```

他可能就不会了

大模型因为参数更多、训练数据更多，可能学到更复杂的关系：

* Redis 为什么快
* MySQL 为什么适合持久化
* 缓存一致性有什么问题
* 高并发下会出现缓存击穿、穿透、雪崩
* 工程上如何设计读写链路

所以大模型强，不只是“知道更多词”，而是更能把知识组织起来

### 3. 训练数据是什么？

大模型训练时会**阅读大量文本**，比如：书籍、论文、网页、代码、文档、问答、对话、数学题、编程题等资料

他的训练过程大致是：

1. 给模型一段文本
2. 让它预测后面应该出现什么
3. 预测错了就调整参数
4. 重复很多很多次

训练完以后，模型就学到了大量语言模式和知识结构

所以你问它问题时，它不是去数据库里查一条固定答案，而是根据当前上下文生成一个最可能合理的回答

### 4. 大模型和普通程序的区别是什么？

普通程序是人写的规则：

```text
如果用户输入 A，就执行 B
如果输入 C，就执行 D
```

大模型是训练学习出来的：

```text
看到这个上下文
根据学到的规律
生成最合适的后续内容
```

所以**普通程序更像“规则机器”，大模型更像“经验系统”**

它不是完全理解世界，也不是简单查表，而是在海量数据训练后，形成了一套复杂的模式判断能力

总结一下：

> **大模型不是一个知识库，而是一个经过海量文本训练后，能根据上下文组织知识、理解意图、生成答案的概率推理系统**



## 二、Transformer——LLM常用的底层架构

首先有个基本认知：

> **Transformer 是一种让模型在读一段文字时，能够同时看见所有词，并判断“哪些词和当前词最相关”的架构**

它之所以能支撑今天的 LLM，核心不是“记住了很多句子”，而是学会了：**在上下文里找关系、聚合信息、预测下一个词**

### 1. 传统模型的问题：一句话要一个词一个词读

早期文本处理，RNN、LSTM这类结构，会像人一样从左到右读句子：

```text
我 -> 今天 -> 去 -> 北京 -> ...
```

问题是：如果句子很长，前面的信息传到后面会越来越弱

比如：

```text
Vect虽然昨天没睡好，今天早上还迟到了，但是他在面试中表现得非常自信
```

需要理解“他”是谁，那么就要回头去找“Vect”，传统结构做这种长距离关联比较吃力

而 Transformer 的想法是：**别一个词一个词传了，所有词同时摊开，让每个词自己去找和它相关的词**



### 2. Transformer 最核心的机制：注意力机制

注意力机制可以理解成：**每个词都会问：在这句话里，我最应该关注谁？**

比如：

```text
Vect把苹果放进书包，因为它太重了。
```

这里的“它”指什么？可能是“苹果”，也可能是“书包”，模型会根据上下文判断：

```text
它 -> 更关注 苹果 / 书包
```

再比如：

```text
Vect把苹果放进书包，因为它太小了。
```

这时“它”更可能指“书包”

所以，**注意力不是死规则，而是根据上下文动态判断关系**

LLM 的很多“理解能力”，底层都来自这种能力：

- 代词指代：他、它、这个、那个

- 语义关联：谁做了什么

- 长文本理解：前文提到的信息，后文还能用上

- 代码理解：变量在哪里定义，在哪里使用

- 推理链条：当前结论依赖哪些前提

### 3. Self-Attention

自注意力机制，意思是**一段文本内部的每个词，都和这段文本里的其他词互相看一遍**

比如：

```text
我 喜欢 用 Go 写 后端 服务
```

模型不是简单地按顺序读，而是让每个词都去看其他词：

```text
Go -> 写、后端、服务
服务 -> 后端、Go、写
我 -> 喜欢、写
```

这样模型就能知道：

* “Go”在这里是编程语言，不是英文单词 go
* “后端服务”是一组相关概念
* “我喜欢用 Go 写后端服务”表达的是技术偏好

这就比单纯顺序读取强很多



### 4. Multi-Head Attention：多个视角一起看

一个注意力机制只能从一个角度看问题，但语言很复杂，一个词和其他词之间可能有很多种关系，比如：

```text
Vect昨天在公司写完了 Go 项目，然后提交到了 GitHub。
```

模型可能需要同时关注：

* 语法关系：谁写完了项目？
* 时间关系：昨天发生了什么？
* 动作关系：写完、提交
* 技术关系：Go 项目、GitHub
* 指代关系：然后是谁提交？

所以 Transformer 不只用一个注意力，而是用多个注意力头

可以把它理解成：

**一群分析人员同时看同一句话，每个人关注不同角度，最后把结果合起来**



### 5. 词本身怎么进入模型？ Embedding

模型不能直接理解文字，它看到的不是“苹果”“Go”“后端”，而是一堆数字表示

Embedding 的作用就是：**把词转换成模型能处理的向量表示**

这里的向量是一组浮点数组成的数组，本质是**高维空间中的坐标，语义相似的词，在空间中的位置就离得近**

举个例子：如何用数字量化一个“人”？

假设使用四个指标来打分（0到100）：

`[学历,成就,身高,体重]`

那么张三可能为：`[80, 70, 175, 90]` 李四可能表示为：`[10, 85, 180, 20]`

这组数字就是张三和李四的 Embedding，通过计算这两组数字的距离，就能知道张三和李四是不是同一类人

这些数字不是人工写死的，而是模型在大量训练中学出来的

所以 LLM 不是真的“懂汉字”，而是学到了文字背后的统计关系和语义结构



### 6. 位置编码：否则模型不知道词的顺序

elf-Attention 有一个特点：它会同时看所有词，这很好，但也带来一个问题：

如果没有额外信息，模型不知道词的顺序，比如：

```
狗咬人
人咬狗
```

词一样，但意思完全不同，所以 Transformer 要加入“位置编码”，告诉模型：

```
这个词在第 1 个位置
这个词在第 2 个位置
这个词在第 3 个位置
```

可以理解成：**Embedding 告诉模型“这个词是什么”，位置编码告诉模型“这个词在哪里”**，两者合起来，模型才知道完整含义

### 7. Feed Forward：每个词自己再加工一下

注意力机制负责让词和词之间交流。但交流完之后，每个词还需要自己加工、提炼一下。

这一步叫 Feed Forward Network。可以把它理解成：

**注意力负责收集上下文信息，Feed Forward 负责消化这些信息。**

比如模型看到：

```
Go 的 goroutine 很轻量
```

注意力让“goroutine”关注“Go”“轻量”，而 Feed Forward 再把这些信息加工成更高级的理解：

```
这里在讲 Go 并发模型的特点
```

### 8.  Decoder-only：为什么 GPT 类模型能生成文字？

 Transformer 有几种结构：

* Encoder-only：适合理解，比如 BERT
* Encoder-Decoder：适合翻译，比如早期机器翻译模型
* Decoder-only：适合生成，比如 GPT 系列、Gemini系列

LLM 大多是 Decoder-only，它的核心任务非常朴素：

**根据前面的内容，预测下一个最可能出现的词**

比如：

```
今天天气很好，我想去
```

模型可能预测：

```
公园
跑步
散步
学校
```

然后选一个词输出，输出后，再把新词接回上下文，继续预测下一个词

所以生成过程本质上是：

```
预测一个词 -> 加到上下文 -> 再预测下一个词 -> 再加进去
```

最后就形成了一整段回答

### 9. LLM 的整体工作流程

![llm工作流程](20260713203415196.png)

通俗来讲，Transformer 像一个读书很快的分析团队，面对一段文字：

* Embedding：先把每个词翻译成模型能懂的表示
* 位置编码：标记每个词的位置
* Self-Attention：每个词去找相关词
* Multi-Head：多个角度同时分析
* Feed Forward：把信息进一步加工
* 多层堆叠：从浅层语法走向深层语义
* Decoder：根据已有上下文不断生成下一个词

最终，LLM 看起来像是在“理解和回答”，底层其实是：

**在巨大语料中学到语言、知识和关系模式后，基于当前上下文一步步生成最合适的后续内容**

总结一下：

> **Transformer 的本质，是用注意力机制在上下文中动态建立关系; LLM 的本质，是基于这些关系不断预测下一个 token**

## 三、大模型的能力

用一句话概括：**基于上下文，理解你的意图，组织已有知识，并生成符合要求的内容**

### 1. 文本理解与生成能力

这是最基本也是最核心的能力，能够理解你输入的自然语言，并生成连贯、有逻辑的回答

大模型先要能读懂人类语言，再用人类能理解的方式表达出来

### 2.推理与逻辑分析

模型不只是复述知识，而是能根据已有信息做分析、判断和推导。但是要注意：**大模型的推理不是绝对可靠的证明，它更像是基于知识和上下文做出的高质量判断**，所以，**推理能力让大模型从“会说知识”变成“会分析问题”**

### 3. 代码理解和生成

这个就不过多赘述

### 4. 工具调用

工具调用是大模型从“只会回答”升级成“能执行任务”的关键。单纯聊天时，大模型只能基于已有知识回答。但有了工具，它可以做更多事，比如：

```text
搜索最新资料
读取 GitHub 仓库
运行代码
分析文件
生成图片
创建文档
调用数据库
操作浏览器
设置定时任务
```

这个能力很重要，弥补了原先大模型的几个短板：

| 大模型短板     | 工具如何弥补             |
| -------------- | ------------------------ |
| 不知道最新信息 | 用搜索、网页、GitHub     |
| 不能运行代码   | 用终端、编译器、测试     |
| 不能看本地文件 | 用文件读取工具           |
| 不能执行动作   | 用日历、邮件、自动化工具 |
| 容易凭空猜     | 用真实数据验证           |

### 5. 多模态理解

多模态理解是指大模型不只处理文字，还能理解图片、截图、表格、音频、视频等信息。

以前的语言模型主要看文本。现在的多模态大模型可以看：

```text
图片
截图
代码截图
网页截图
表格
PDF
流程图
架构图
视频帧
语音内容
```

多模态的本质是：**把不同形式的信息都转成模型能处理的内部表示，再和文字一起理解**



总结一下:
> 最底层的是文本能力。
> 在文本能力之上，发展出推理、代码和多模态能力。
> 再加上工具调用，大模型就从一个聊天模型变成了一个可以协作完成任务的智能系统。
